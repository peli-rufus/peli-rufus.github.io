Linux OS — Forensics Log & Artefact Value Matrix

Quick-reference for incident response and forensic investigation of Linux systems.

Key notes: Log paths vary by distribution — Debian/Ubuntu uses /var/log/auth.log and /var/log/syslog; RHEL/CentOS/Rocky uses /var/log/secure and /var/log/messages. On systemd-based systems, the primary log store is the systemd journal (journalctl) which may be the only source if rsyslog is not installed. Shell history files are per-user and easily deleted — their absence on an active system is itself suspicious. On a compromised system with a rootkit present, standard system tools (ps, ls, ss, lsof) cannot be trusted — use statically compiled or trusted binaries from read-only media. Collection tool: UAC (Unix-like Artifact Collector, tclahr/uac). Memory acquisition: LiME (Loadable Kernel Module).

Authentication & Session Logs
ArtefactPath (Debian / RHEL)Persists reboot?Has timestamps?Key forensic valueCaveats
auth.log / secure auth /var/log/auth.log (Debian)
/var/log/secure (RHEL)		 Yes Yes — per event All SSH logins (success and failure), sudo usage, PAM events, su switches, cron authentication. Primary authentication audit source. Source IP visible on SSH events. Rotated — check auth.log.1, auth.log.2.gz for older events. SSH brute force leaves high-volume failure entries before a success. Sudo usage shows exact commands with user context.
wtmp session /var/log/wtmp Yes Login/logout timestamps Binary record of every login session — user, terminal, source IP, login time, logout time. Read with last command. Provides session duration. Binary format — do not cat directly. Parse with last -f /var/log/wtmp. Can be modified or zeroed by attacker — absence of expected sessions is an IOC.
btmp auth /var/log/btmp Yes Yes Binary log of all failed login attempts. Read with lastb. Shows brute force attempts including targeted usernames and source IPs. Binary format — parse with lastb -f /var/log/btmp. Often rotated as btmp.1. High entry count from single IP = brute force campaign.
lastlog auth /var/log/lastlog Yes Last login per user Records the last login time and source for every user account on the system. Read with lastlog. Quickly shows dormant accounts that suddenly became active. Binary sparse file indexed by UID — one entry per user. Never-logged-in accounts show as **Never logged in**. Backdoor account that suddenly shows a login is a strong IOC.
utmp (live sessions) session /var/run/utmp No — volatile Yes Currently logged-in users at time of collection. Read with who or w. Shows active terminals and source IPs. Volatile — only useful on a live system. Lost after reboot. On a compromised system, rootkits can hide sessions from utmp while keeping them active.
faillog auth /var/log/faillog Yes Yes Failed login counts per user with timestamps. Accounts with high failure counts followed by a success indicate successful brute force. Binary format — parse with faillog command. Not present on all distributions.
System & Kernel Logs
ArtefactPathPersists reboot?Has timestamps?Key forensic valueCaveats
syslog / messages system /var/log/syslog (Debian)
/var/log/messages (RHEL)		 Yes Yes General system messages from kernel and services. Service start/stop events, cron job execution records (cron logs here on many distros), network configuration changes, hardware events. Cron job executions logged here by default — shows what ran and when. Service crash/restart events useful for detecting killed security tools. Rotated — check syslog.1 and compressed archives.
kern.log / dmesg kernel /var/log/kern.log
/var/log/dmesg
dmesg command (live ring buffer)		 kern.log yes, dmesg volatile Yes Kernel messages — module load/unload events, hardware events, network driver activity. LKM (Loadable Kernel Module) rootkit insertion leaves a trace here. OOM killer events indicate memory pressure. Kernel module insertion (insmod/modprobe) logged here. Suspicious module name or path = LKM rootkit indicator. dmesg ring buffer is volatile — collect immediately on live response. Rootkits may suppress kernel messages.
systemd journal system /var/log/journal/ (persistent)
/run/log/journal/ (volatile)		 Only if Storage=persistent in journald.conf Yes — microsecond precision On systemd systems, the journal is the primary log for all services, units, and kernel messages. Binary format — query with journalctl. Captures stdout/stderr of every service unit. Harder to partially tamper than text logs. Binary format — cannot be tampered with a text editor. Collect with journalctl --no-pager > journal.txt or copy the binary journal files. Persistence must be explicitly enabled — default is volatile on many distros.
audit.log (auditd) audit /var/log/audit/audit.log Yes Yes — epoch timestamps If auditd is configured, this is the highest-fidelity log on Linux. Captures syscalls, file access, process execution, network connections, user/group changes, and privilege escalation — all with user context (UID/AUID). AUID (audit UID) survives su/sudo and tracks the original login session. Only as useful as the audit rules configured. Default rules are minimal. AUID (loginuid) is the key field — traces activity back to the original SSH session even after sudo. Parse with ausearch and aureport. aureport -au for authentication, aureport -x for executions.
boot.log system /var/log/boot.log Yes Yes Service start/stop events during system boot. Records which services were enabled and their startup status. Useful for confirming whether persistence mechanisms executed at boot. Not present on all distributions. Systemd-based systems may rely on the journal for boot-time service events instead.
Shell History & User Activity
ArtefactPathPersists reboot?Has timestamps?Key forensic valueCaveats
bash_history execution ~/.bash_history
/root/.bash_history		 Yes No (unless HISTTIMEFORMAT set) Commands typed by the user in bash. Most forensically valuable user activity artefact on Linux. Reveals attacker tooling, download commands, lateral movement, and privilege escalation attempts. Easily deleted or cleared with history -c or unset HISTFILE. Absence on an active system is suspicious. HISTTIMEFORMAT env var adds timestamps — check if set. Commands prefixed with a space may be excluded (HISTCONTROL=ignorespace). Check all user home dirs including /root.
zsh_history / fish_history execution ~/.zsh_history
~/.local/share/fish/fish_history		 Yes zsh: optional, fish: yes Equivalent of bash_history for zsh and fish shells. zsh_history can include timestamps if extended_history is set. fish_history includes timestamps by default. Check which shell each user account uses — /etc/passwd lists the default shell. Attackers may switch shells specifically to avoid bash history logging.
.viminfo / .lesshst user activity ~/.viminfo
~/.lesshst		 Yes Partial .viminfo records files recently opened in vim, recent search patterns, and yanked content — can reveal config files the attacker edited. .lesshst shows files paged with less. Frequently overlooked. Can reveal exactly which files were viewed or edited even when bash_history is cleared.
recently used files (.recently-used.xbel) user activity ~/.local/share/recently-used.xbel Yes Yes GTK recently used files list — files opened by GUI applications. XML format. Reveals documents, scripts, or archives opened via file manager or GUI apps. Only present on systems with a desktop environment. Not relevant for headless servers.
sudo log entries privilege /var/log/auth.log or /var/log/secure Yes Yes Every sudo command with the invoking user, the target user (usually root), the working directory, and the exact command run. Provides full privilege escalation audit trail. grep -i sudo /var/log/auth.log to extract quickly. sudo commands with env_reset bypassed (NOPASSWD entries in /etc/sudoers) are a persistence/privilege escalation indicator.
File System & Inode Artefacts
ArtefactPath / CommandSurvives deletion?Timestamps available?Key forensic valueCaveats
Inode timestamps (mtime / ctime / atime) filesystem stat <file>
find / -newer /tmp/ref_file		 Until block reuse mtime / ctime / atime mtime = last content modification. ctime = last metadata change (cannot be set back by normal tools — useful for detecting touch -t tampering). atime = last access (often disabled for performance). ctime is the closest Linux equivalent to a tamper-detection timestamp. atime is often mounted with noatime or relatime — do not rely on it. ctime cannot be set by users but is updated on permission/owner changes. Unlike Windows, Linux has no creation time (birthtime) on most filesystems — ext4 stores it in crtime but tools rarely expose it.
ext4 filesystem journal filesystem debugfs -R 'logdump -a' /dev/sdX
extundelete		 Short window Partial Records recent metadata transactions — file creation, rename, deletion. Can recover recently deleted files and reveal file names that have been unlinked. Equivalent to Windows $LogFile. Journal wraps quickly on busy systems. Must work on offline/unmounted filesystem for accuracy. Tool: extundelete, debugfs. SSD TRIM may prevent recovery.
find — recently modified files filesystem find / -mtime -1 -type f 2>/dev/null
find /tmp /var/tmp /dev/shm -type f		 Until deletion Yes Quickly surfaces recently created or modified files. Focuses on attacker staging directories (/tmp, /var/tmp, /dev/shm). New files in /etc, /usr/bin, /lib are high suspicion. /dev/shm is a memory-backed tmpfs — files here never touch disk and are lost on reboot. Attacker payloads stored here leave no filesystem artefact after reboot. Collect immediately.
SUID / SGID binaries privilege find / -perm -4000 -type f 2>/dev/null
find / -perm -2000 -type f 2>/dev/null		 Yes ctime reflects when SUID was set New SUID binaries are a classic privilege escalation and persistence mechanism. An attacker with root who sets /bin/bash -p SUID creates a persistent root shell. Compare against known-good SUID list. Baseline comparison is essential. All legitimate SUID binaries are well-known — any unexpected path (especially /tmp, /home, /var) is malicious. ctime of the binary will show when the SUID bit was set.
Hidden files and dot-directories filesystem ls -la /tmp /var/tmp /dev/shm /home/*
find / -name ".*" -type f 2>/dev/null		 Until deletion Yes Attacker tools and staging data frequently hidden with leading dot in filename or placed in world-writable directories. Double-dot directories (.. with space) are a common obfuscation technique. Check /tmp, /var/tmp, /dev/shm, /var/www, and all user home directories. Files named with spaces or unusual Unicode are anti-forensics indicators.
Deleted but open file descriptors filesystem ls -la /proc/*/fd | grep deleted
lsof | grep deleted		 Lost on reboot Via /proc Files deleted from the filesystem but still held open by a running process remain accessible via /proc/PID/fd/. Common technique — malware deletes its own binary after execution to hide it from disk scans, but content readable via /proc. Volatile — only on live system. Content retrievable with cp /proc/PID/fd/N /tmp/recovered_binary. This is a first-priority live response action before any process is killed.
Process & Memory Artefacts (Live Response)
ArtefactSourceVolatile?Key forensic valueCaveats
/proc filesystem execution /proc/<PID>/cmdline
/proc/<PID>/exe
/proc/<PID>/maps
/proc/<PID>/environ
/proc/<PID>/net/tcp		 Yes — volatile cmdline = full command line including arguments. exe = symlink to the binary on disk (check if pointing to deleted file). maps = memory mapped regions including injected libraries. environ = environment variables including LD_PRELOAD. net/tcp = network connections per process. All /proc entries are volatile and disappear when the process ends or system reboots. Collect all of /proc for key PIDs immediately. exe pointing to a path marked (deleted) = binary was running from a file that has since been removed — recover via fd.
Process list anomalies execution ps aufwx
pspy64 (unprivileged process monitor)		 Volatile Unexpected processes, processes with misleading names (kworker, systemd-X), processes running from /tmp or /dev/shm, processes with no controlling terminal (detached backdoors). On a rootkit-compromised system, ps output cannot be trusted — rootkit hooks hide processes. Use pspy64 (unprivileged, reads /proc directly) or compare /proc directory listing against ps output. Discrepancy = hidden process (rootkit present).
Network connections network ss -tunlp
netstat -tulpn
/proc/net/tcp /proc/net/tcp6		 Volatile Active network connections with associated PIDs and listening ports. Unexpected outbound connections to unusual IPs = C2 beaconing. Unexpected listening ports = bind shell or backdoor listener. ss/netstat can be hidden by rootkits — cross-reference with /proc/net/tcp (harder to hide). Compare listening ports against expected service baseline. Connections to loopback on unusual ports indicate local inter-process C2 (e.g. reverse proxy).
Memory dump (LiME) memory insmod lime.ko "path=/mnt/evidence/mem.lime format=lime" Volatile Full RAM acquisition. Enables Volatility3 analysis — process listing, network connections, kernel module enumeration, string searching. The only reliable source when rootkit hides data from OS tools. LiME is a kernel module — insertion modifies the running system slightly. Must be compiled for the exact kernel version of the target. Tool: LiME (Volatility compatible). Analyse with Volatility3 linux plugins: linux.pslist, linux.netstat, linux.lsmod, linux.bash.
Loaded kernel modules kernel lsmod
/proc/modules
/proc/kallsyms
/sys/module/		 Volatile List of loaded kernel modules. LKM rootkits load as kernel modules and may hide from lsmod. Compare lsmod output against /proc/modules listing — discrepancy = hidden LKM rootkit. LKM rootkits use list_del() to remove themselves from the module linked list shown by lsmod, but traces may remain in /proc/kallsyms (exported symbols) and kern.log (load event). Tools: unhide, Tracee (eBPF-based).
User & Account Artefacts
ArtefactPathPersists reboot?Key forensic valueCaveats
/etc/passwd accounts /etc/passwd Yes All user accounts — username, UID, GID, home directory, default shell. UID 0 accounts beyond root = backdoor account. Accounts with /bin/bash shell but no legitimate purpose = suspicious. Accounts with UID matching a legitimate account but different name = collision attack. World-readable. Check for non-standard UID 0 entries (multiple root-equivalent accounts). Accounts with home directories in /tmp or /dev are strong IOCs. Compare against a known-good baseline.
/etc/shadow accounts /etc/shadow Yes Password hashes, password age, expiry. Recently changed password hashes = attacker may have set a new password. Accounts with no password (empty hash field) = unauthenticated access possible. Root-readable only. Hash format prefix identifies the algorithm: $1$=MD5 (weak), $5$=SHA-256, $6$=SHA-512, $y$=yescrypt (modern). Offline cracking with hashcat on extracted hashes.
/etc/sudoers and sudoers.d privilege /etc/sudoers
/etc/sudoers.d/*		 Yes Defines who can run what commands as root. NOPASSWD entries for ALL commands = root access without password. Unexpected user entries = privilege escalation persistence. Attacker often drops a file in /etc/sudoers.d/ rather than editing the main sudoers file — easier to hide. Check both locations. visudo validates syntax — direct file edits may leave invalid entries.
SSH authorized_keys persistence ~/.ssh/authorized_keys
/root/.ssh/authorized_keys		 Yes Public keys authorised for passwordless SSH login. Attacker-planted key provides persistent backdoor access without any log entry showing a password attempt. Check for all users with SSH access. Look for unusual key comments (email fields) or keys with unusual options (command= restriction, no-pty). Immutable attribute (chattr +i) used by advanced attackers to prevent removal — check with lsattr.
/etc/ssh/sshd_config modifications persistence /etc/ssh/sshd_config Yes SSH daemon configuration. PermitRootLogin yes = direct root SSH enabled. PasswordAuthentication yes = brute force possible. AuthorizedKeysFile pointing to an unusual path = backdoor key loading mechanism. ctime of the file shows when it was last changed. Compare against distro default or known baseline. Attackers frequently enable PermitRootLogin to ensure persistent access after planting a root SSH key.
/etc/group accounts /etc/group Yes Group memberships. Unexpected addition of a user to the sudo, wheel, or docker group = privilege escalation. Docker group membership = effective root (docker escape). docker group = root equivalent on most systems. Any non-administrative user in sudo/wheel/docker should be investigated. Cross-reference with auth.log for when the change was made.

/etc/bash.bashrc		 System-wide = yes, user = no Yes Look for appended lines at the bottom of shell init files — attackers typically append rather than overwrite. LD_PRELOAD / dynamic linker hijack persistence /etc/ld.so.preload
/etc/ld.so.conf
LD_PRELOAD env var in shell init files		 ld.so.preload = yes, env var = no Yes /etc/ld.so.preload not set by default — any entry is suspicious. Check with ldd <binary> to see injected libraries. Cross-reference /proc/<PID>/maps for unexpected .so files loaded into processes. Malware: HiddenWasp, Symbiote, Azazel all use this technique. T1574.006. LKM rootkits may protect /etc/ld.so.preload from modification or viewing — standard cat will return empty even when the file has content. Use statically compiled tools or memory forensics. Cannot be detected if rootkit is actively hiding it.
LKM (Loadable Kernel Module) rootkit persistence /lib/modules/$(uname -r)/kernel/
/proc/modules
/proc/kallsyms
lsmod vs /proc/modules comparison		 Yes (if loaded at boot) Yes Discrepancy between lsmod output and /proc/modules content = hidden module. Unexpected symbols in /proc/kallsyms. kern.log shows module load event. Tool: unhide, Tracee (eBPF). Module loaded from unusual path (not /lib/modules/) is strong IOC. Most dangerous persistence — operates at kernel level, can hide all evidence. Standard tools completely unreliable once LKM rootkit is active. Trusted analysis requires booting from read-only media or memory forensics. Known rootkits: Diamorphine, Reptile, Reptile2.
SysV init / rc.local persistence /etc/init.d/*
/etc/rc.local
/etc/rc*.d/		 Yes Yes rc.local executes at boot. init.d scripts define services. Malicious content appended to rc.local or new init.d scripts. File ctime shows modification time. T1037.004. Common on older systems and some distros (Debian still supports rc.local). systemd systems may emulate rc.local via rc-local.service — still executes. Check file content and ctime.
MOTD (Message of the Day) scripts persistence /etc/update-motd.d/*
/etc/motd		 No — root required Yes update-motd.d scripts execute as root on every SSH login. Malicious script here runs with root privileges every time any user logs in via SSH. Rarely audited. Not present on RHEL derivatives. Particularly effective because it executes on every SSH session including automated ones. Any executable in /etc/update-motd.d/ that does more than display text is suspicious.
udev rules persistence /etc/udev/rules.d/*
/lib/udev/rules.d/*
/run/udev/rules.d/*		 Yes Yes udev rules trigger on hardware events. Malicious rule executes a payload when any device is connected or on network interface changes. Rarely monitored. Used by Sedexp malware. T1546 (uncategorised subtechnique). Rarely seen in the wild — highly suspicious if any rule contains RUN+= pointing to a non-standard binary. udevadm monitor on live system shows rule triggers.

SSH known_hosts network ~/.ssh/known_hosts
/etc/ssh/ssh_known_hosts		 Persistent Fingerprints of SSH servers the user has previously connected to. Reveals lateral movement targets — hosts the attacker connected to from this machine. Each entry proves an outbound SSH connection was made. Per-user. Host entries added at first connection. Attacker may use StrictHostKeyChecking=no to avoid adding entries (check bash_history for this flag). Global known_hosts in /etc/ssh/ shows system-wide connections.
resolv.conf modifications network /etc/resolv.conf Persistent DNS server configuration. Attacker may change to rogue DNS to facilitate C2 resolution. Adding 8.8.8.8 to bypass internal DNS is a common attacker action to reach external C2. ctime shows when modified. On systemd-resolved systems, this may be a symlink — check the actual configuration file. Modification of /etc/hosts to add attacker-controlled IP mappings is a related technique.
/etc/hosts modifications network /etc/hosts Persistent Static DNS overrides. Attacker may add entries to redirect legitimate domains to attacker infrastructure (C2 domain pointing to internal IP) or to reach C2 hosts by name without DNS. Compare against known-good baseline. Any entry pointing to an external IP for an internal domain name, or vice versa, is suspicious.
iptables / nftables rules network iptables -L -n -v
nft list ruleset
/etc/iptables/rules.v4		 Persistent if saved Active firewall rules. Attacker may add rules to allow inbound connections to backdoor ports, or to block outbound connections from security tools. New ACCEPT rules for unusual ports are IOCs. Unsaved rules (memory only) are lost on reboot. Check both current rules (iptables -L) and saved rules files (/etc/iptables/). New INPUT ACCEPT rules added without corresponding service = backdoor port.

/var/log/nginx/access.log

Source IPs, user agents, request paths, response codes. Collection Tools & Triage Approach
ToolWhat it collectsWhen to useNotes
UAC (Unix-like Artifact Collector) 473+ Linux artefacts — logs, process info, network state, user accounts, cron, systemd units, SSH keys, shell history, /proc, open files, kernel modules First step on any suspected Linux compromise. Single portable shell script — no installation required. Uses target system binaries by default — if rootkit is present, results may be unreliable. Include statically compiled trusted binaries via UAC's /bin directory override for rootkit scenarios. Output is a collection of text files importable into Cyber Triage.
LiME (Linux Memory Extractor) Full physical memory acquisition to file or network socket When memory forensics is needed — fileless malware, rootkit analysis, credential extraction from RAM. Kernel module — must be compiled for exact target kernel version (uname -r). Minimal memory footprint during acquisition. Output analysable with Volatility3 linux plugins.
pspy64 Real-time process monitoring without root — reads /proc directly Live response when you need to watch for cron/systemd executions or spot processes a rootkit may be hiding from ps. Unprivileged — does not require root. Shows new process creation events including those hidden from ps. Useful for confirming cron job execution timing.
unhide Finds processes and ports hidden by rootkits by comparing multiple data sources When LKM rootkit is suspected — compare ps, /proc, syscall enumeration, and brute-force PID scanning. May itself be hooked by a sufficiently advanced rootkit. Best run from read-only media with statically compiled binary. unhide sys for process hiding, unhide tcp for port hiding.
Volatility3 (linux plugins) Process list, network connections, kernel modules, bash history from memory, environment variables, kernel symbol hooks After LiME memory acquisition. Essential for rootkit analysis and fileless malware investigation. Key plugins: linux.pslist, linux.netstat, linux.lsmod, linux.bash, linux.malfind, linux.check_syscall (detect syscall hooking). Requires a matching kernel symbol file (ISF profile) for the target kernel version.
auditd + ausearch / aureport Syscall-level audit — file access, process execution, network, privilege changes (if configured) When auditd is running on the target. Highest fidelity source if audit rules cover the relevant activity. aureport -au for auth events, -x for executable events, -f for file access. ausearch -k <key> for specific rule matches. AUID field traces events back to original SSH session even after sudo. Not present by default on all distros.
