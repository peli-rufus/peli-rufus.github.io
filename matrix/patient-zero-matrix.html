How to use this tab: for each step in the matrix, one primary tool is picked out and given a concrete, hands-on procedure — the query to run, what to look for, and what it feeds into next. Treat this as a starting playbook, not the only valid option.

Phase 1 — External Recon
01
Infostealer log search → Hudson Rock (Cavalier Free Tools)

Go to hudsonrock.com/free-tools and run the "Cybercrime Intelligence Search" against your root domain and any key employee emails. The results table returns compromised machine entries tagged with infection date, malware family (Redline/Lumma/Vidar/etc.), and which of your URLs appeared in the saved-credential data.

Watch for: any infection date that predates your known breach window — that's your dormancy period, and the machine behind it is your patient-zero candidate.

02
External exposure sweep → Shodan

Search org:"Your Company Name" or hostname:yourdomain.com, then narrow with port:3389, port:3306, or http.title:"phpMyAdmin". Shodan's "Ports" and "Top Organizations" panels give you a quick inventory of what's actually reachable from the internet.

Watch for: any admin panel, database port, or RDP service you didn't know was exposed — cross-check it against the subdomains from Step 03.

03
Subdomain enumeration → crt.sh

Query crt.sh/?q=%.yourdomain.com, export the unique Common Name / Subject Alternative Name entries, then resolve which are actually live with httpx or a quick DNS sweep.

Watch for: dev/staging/legacy subdomains still resolving — these are the forgotten doors attackers find first (as with AFA's ezeiza subdomain).

04
Dark web / forum monitoring → Flare

Set up a brand/domain monitor for your company name and domain. Flare's daily digest surfaces forum posts, Telegram messages, and paste-site hits automatically, so you don't manually browse criminal forums yourself.

Watch for: "access for sale" listings naming your subdomains, or database sample posts — these usually appear before wider media coverage.

Phase 2 — Internal Correlation
05
Auth log correlation → Splunk / Sentinel

Build a query filtering sign-in events to the date window around the Step 01 infection date, excluding known corporate IP ranges/ASNs. In Sentinel: SigninLogs | where TimeGenerated between (start..end) | where Location != "expected".

Watch for: a login from an unfamiliar ASN or country shortly after the infection date but before the public breach — that gap is your dwell time.

06
EDR / AV history check → CrowdStrike Falcon

In the Detections dashboard, filter by malware family name (Redline, Lumma, Vidar) and the date range from Step 01. Check each detection's disposition — many organizations have the alert but it was auto-resolved or never triaged.

Watch for: a detection marked "resolved" with no follow-up ticket — that's usually the missed signal.

07
Browser credential audit → Intune / Jamf (MDM)

Pull the password-manager policy compliance report across managed devices. Cross-reference any device with browser-saved-password permissions enabled against your list of admin-panel URLs.

Watch for: password reuse across multiple saved entries on the same device — a strong predictor of lateral admin access if that device is ever compromised.

08
Email gateway review → Proofpoint / Mimecast

Search inbound mail logs to the flagged account for the 2–3 weeks before the infection date, filtering for attachment types commonly used in stealer delivery: .iso, .lnk, .js, password-protected archives.

Watch for: a single suspicious message that correlates tightly with the infection timestamp — that's your likely initial-access vector.

Phase 3 — Attribution
09
Identity cross-reference → Entra ID / Active Directory

Look up the flagged username or device owner, and pull group memberships and role assignments directly from the directory.

Watch for: an account with standing admin rights to more systems than its job function requires — a common root cause, independent of the infection itself.

10
Password reuse verification → Manual test, in-scope only

Only within a signed authorization/pentest scope, attempt the leaked credential against the live panel it was tied to. Document with a timestamped screenshot, then rotate the credential immediately after confirming.

Watch for: this step is the only one on the matrix that touches a live production credential — never run it without written sign-off.

11
Timeline reconstruction → Spreadsheet / i2 Analyst's Notebook

Plot every dated event from Steps 01–10 on one timeline: infection → log detection → dormancy → first anomalous login → exfiltration → public disclosure.

Watch for: the length of the dormancy gap — this is usually the single most persuasive number for justifying budget for continuous leak monitoring (Step 23).

Phase 4 — Malware & Infrastructure
12
Sample analysis → Any.Run

Upload the recovered binary to an interactive sandbox. Watch the network tab for C2 callbacks and the process tree for injection or persistence technique.

Watch for: the C2 domain/IP — this feeds directly into Step 13's infrastructure pivot.

13
C2 / infrastructure pivoting → VirusTotal Graph

Pivot from the C2 IP/domain to shared SSL certificates, hosting providers, or historical WHOIS registrant data to surface other infrastructure tied to the same actor.

Watch for: overlap with infrastructure named in other public breach reports — a strong campaign-correlation signal.

14
IOC extraction & sharing → MISP

Create an event, attach hashes/domains/IPs as attributes, tag with a TLP marking, and publish to your internal detection stack or sector ISAC community.

Watch for: attribute overlap with existing MISP events — may mean you're not the first victim of this exact infrastructure.

15
Detection engineering → YARA + Sigma

Write a YARA rule on unique strings/imports from the sample, and a Sigma rule on the log-based behavior pattern (e.g. bulk browser credential-file access). Deploy both to EDR/SIEM.

Watch for: false-positive rate during a burn-in period before enabling blocking mode, not just alerting.

Phase 5 — Actor Intelligence
16
Threat actor profiling → Recorded Future

Search the actor's forum/Telegram handle for post history, reputation score, and prior victims or listings.

Watch for: a pattern of "sell access" vs "leak and shame" behavior — tells you what kind of monetization to expect next.

17
Campaign correlation → Sector ISAC bulletins

Search your sector's ISAC portal for the stealer family name plus your date range to see if peer organizations reported similar activity.

Watch for: a cluster of same-family infections across your sector in the same month — suggests a coordinated or opportunistic campaign, not a targeted attack.

18
Extortion / negotiation monitoring → Legal-coordinated single channel

Designate one point of contact for any actor communication, log everything, and route all decisions through legal counsel before responding.

Watch for: pressure tactics with a countdown or "proof of more data" — do not let this drive an unauthorized unilateral response.

Phase 6 — Validation & Scoping
19
Leak authenticity verification → Manual field-level diff

Compare the leaked sample's field names, hash formats, and record structure against your actual database schema and a few known internal records.

Watch for: mismatched schema or stale records — a sign the actor is padding or recycling an older leak to look more credible.

20
Blast radius / third-party exposure → Vendor access review

Pull the list of vendor/contractor accounts with access to the affected system and check whether any of them share the same credential pattern.

Watch for: a third party that turns out to be the actual entry point — common when a vendor's own device was the one infected.

21
Supply-chain check → Endpoint software inventory

Pull the installed-application list from the infected host via your EDR or asset management tool and check licensing/install source for each entry.

Watch for: cracked/pirated software or unmanaged BYOD status — reshapes both the root-cause narrative and who's liable for remediation cost.

Phase 7 — Post-Incident
22
Rotation verification → IAM last-changed audit

Query IAM for the last-password-changed timestamp on every exposed account and confirm the change happened after the leak was discovered, everywhere the credential was reused.

Watch for: a system that was missed in the first rotation pass — reused passwords rarely live in just one place.

23
Continuous leak monitoring → Hudson Rock / SpyCloud alerts

Set up a recurring scheduled scan (weekly is typical) on your domain, with automatic notification to the security team on any new hit.

Watch for: this is the control that would have caught the original AFA-style dormancy period months earlier — treat it as a standing program, not a one-off check.

24
Regulatory / legal notification tracking → Legal counsel + compliance tracker

Map the exposed data types against applicable breach-notification laws (e.g. GDPR's 72-hour window, relevant state/national laws) together with legal.

Watch for: notification clocks that start from discovery, not from confirmation — don't let technical validation delay the legal clock unnecessarily.

25
Retrospective / control-gap analysis → Post-incident review template

Run a blameless lessons-learned session tying the root cause to one specific missing control (e.g. no MFA on the admin panel, no browser credential policy), with a named owner and remediation deadline.

Watch for: vague action items with no owner — the most common reason the same root cause recurs at the next incident.

Steps 01–04 use only publicly available intelligence and are safe to run against your own organization at any time. Steps 05–11 require standing IR authority or a written, scoped engagement — do not run these against systems you don't own or aren't contracted to assess.
