vCenter / ESXi — Forensics Log & Artefact Value Matrix

Quick-reference guide for incident response and forensic investigation of VMware vSphere environments.

Architecture context: ESXi is a bare-metal Type-1 hypervisor. vCenter Server (VCSA) is a Linux appliance (Photon OS) managing one or more ESXi hosts. Logs exist at three tiers: vCenter (management plane), ESXi host (hypervisor plane), and VM guest (per-VM). ESXi logs are stored on a RAM-backed ramdisk by default — they do NOT survive a reboot unless a persistent scratch location is configured. The vmsyslogd daemon writes logs; killing it (a known attacker technique) stops all local log writing. Always prioritise collecting logs via support bundle (vm-support) or remote syslog before any reboot. Collection tool: DFIR4vSphere (ANSSI-FR, PowerShell/PowerCLI). Parser: QELP (Stroz Friedberg).

vCenter triage priority: 1. VI Events (structured API audit — broadest coverage) → 2. vpxd.log (main vCenter log) → 3. sso-events.log (authentication) → 4. vpostgres log (if DB credential extraction suspected) → 5. journalctl ⚠ secondary — only when OS-level breakout is suspected, a VMware service crashed with no app-log record, or VCSA is being treated as a compromised Linux host.

vCenter Server (VCSA) Logs — /var/log/vmware/
Log file Path (VCSA) Persists reboot? Retention Key forensic value Threat actor relevance
vpxd.log primary /var/log/vmware/vpxd/vpxd.log Yes Rotated — numbered vpxd-N.log Main vCenter log. All vSphere client connections, API calls, internal tasks/events, vpxa communication with ESXi hosts, DB operations. First log to examine. Attacker logins via vSphere client, API-based VM operations (snapshot, power off, export), privilege changes all appear here. vCenter database (VCDB) queries logged here including credential extraction attempts.
vpxd-svcs.log api /var/log/vmware/vpxd-svcs/vpxd-svcs.log Yes Rotated vCenter REST API and web services activity. Complements vpxd.log with API-layer details. REST API exploitation (CVE-based attacks against vCenter) will appear here. Malicious Automation via API tokens logged.
sso-events.log / ssoAdminServer.log auth /var/log/vmware/sso/ Yes Rotated SSO authentication events — logins, failures, identity source queries, token issuance. The primary authentication audit trail for vCenter. Brute force against vCenter SSO, stolen session token use, and identity provider manipulation all surface here. Password spraying visible as repeated failures across accounts.
vsphere-ui/catalina.out ui /var/log/vmware/vsphere-ui/logs/ Yes Rotated vSphere Web Client (HTML5) session activity. UI-initiated operations, errors, and connectivity events. Complements vpxd.log for UI-originated attacker actions. Web client exploitation attempts visible here.
vpostgres / postgresql.log database /var/log/vmware/vpostgres/ Yes Rotated PostgreSQL database activity for vCenter. Query logging, transaction events, startup/shutdown. VCDB contains host credentials (vpx_host table). Direct SQL injection or interactive VCDB queries to extract host credentials (SELECT ip_address, user_name, password FROM vpx_host) appear as LOG statements. UNC3886 / Fire Ant used this technique.
VI Events (vSphere API events) api vCenter DB (VCDB) — query via PowerCLI Get-VIEvent Yes Default 30 days (configurable) Structured audit trail of all vSphere events: VM power on/off, snapshot create/delete, host add/remove, permission changes, login/logout. Queryable via PowerCLI. Most comprehensive audit source. Collect with DFIR4vSphere Start-VC_Investigation. Can sometimes retrieve events beyond configured retention. Attacker snapshot creation (for data staging) and VM export visible here.
audit.log (VCSA appliance) auth /var/log/audit/audit.log Yes Rotated Linux-level audit log for the VCSA appliance itself. Root logins, privilege escalation, file access by system processes. If attacker breaks out of vCenter to the underlying Photon OS, this log captures OS-level activity. Treat VCSA as a Linux host in post-exploitation scenarios.
systemd journal (journalctl) system
⚠ SECONDARY SOURCE		 /var/log/journal/ (persistent)
/run/log/journal/ (volatile)		 Yes — Photon OS defaults to persistent Rotated by size OS-level unified log for all VCSA systemd services. Adds value in three specific scenarios only: (1) attacker broke out of vCenter to the underlying Photon OS shell, (2) a VMware service (vpxd, vpostgres) crashed abruptly and the application log has no record, (3) VCSA itself is being treated as a compromised Linux host. In a standard vCenter investigation — credential abuse, API calls, VM manipulation — vpxd.log and VI Events are richer and should be examined first. Binary format — export with journalctl --no-pager > journal.txt before copying raw files. Useful unit filters: -u vmware-vpxd, -u vmware-vpostgres, -u sshd, -p err (errors only). Do not treat as a replacement for /var/log/vmware/ logs — it is a supplementary OS-layer source. Cross-reference timestamps with vpxd.log and sso-events.log to build the full incident timeline when OS-level activity is suspected.
ESXi Host Logs — /var/log/ (ramdisk — volatile by default)
Log file Path Persists reboot? Key forensic value Threat actor relevance
shell.log execution /var/log/shell.log
/scratch/log/shell.log		 Only if scratch configured Every command typed in the ESXi shell, SSH sessions, and DCUI. Enable/disable events for the shell service itself. Closest equivalent to bash_history on ESXi. Highest forensic value for attacker activity on the host. Ransomware operators typing esxcli, vim-cmd, and crypto commands visible here. Shell enable event itself is a detection trigger.
auth.log auth /var/log/auth.log
/scratch/log/auth.log		 Only if scratch configured SSH and PAM authentication — successful logins, failures, and key-based auth events. Source IP visible. Brute force SSH (multiple failures from same IP), then successful login = strong IOC. Root SSH logins with key (no password failure) = potential backdoor SSH key planted.
hostd.log management /var/log/hostd.log
/scratch/log/hostd.log		 Only if scratch configured Host management service. VM tasks and events (power on/off, snapshot, clone, migrate), vSphere client connections to host directly, vpxa communication. Most operationally rich ESXi log. Direct host connections bypassing vCenter appear here (attacker connecting to host IP directly). VM snapshot creation for staging, VM power operations for disruption all logged. Look for unexpected user accounts or source IPs.
vpxa.log agent /var/log/vpxa.log
/scratch/log/vpxa.log		 Only if scratch configured vCenter Agent on ESXi. Communication between the ESXi host and its managing vCenter. Connection state, task execution instructions received from vCenter. vpxa disconnect/reconnect events show when an ESXi host was detached from or re-added to vCenter. Attacker-induced disconnects visible here. Correlate with vCenter vpxd.log timestamps.
vmkernel.log kernel /var/log/vmkernel.log
/scratch/log/vmkernel.log		 Only if scratch configured VMkernel events — device discovery, storage operations, network stack, VM start/stop at kernel level, hardware errors. Low-level OS activity. NFC (Network File Copy) connections for VM data transfer appear here. Mass VM shutdown events (pre-ransomware encryption) leave a signature. Storage device enumeration during attacker reconnaissance visible.
syslog.log system /var/log/syslog.log
/scratch/log/syslog.log		 Only if scratch configured General system messages. Management service init, watchdog events, scheduled tasks, DCUI interactions. vmsyslogd kill events appear here immediately before the log goes silent — a strong anti-forensics indicator. Watchdog restarts of services post-tampering visible.
vobd.log observations /var/log/vobd.log
/scratch/log/vobd.log		 Only if scratch configured VMkernel observation events. VM migration events (vMotion), storage-related observations, connectivity changes. Unexpected vMotion events (attacker moving VMs for staging or isolation) visible. Correlate with hostd.log and VI Events for confirmation.
fdm.log (HA) cluster /var/log/fdm.log
/scratch/log/fdm.log		 Only if scratch configured Fault Domain Manager — HA cluster events. Host failures, VM restarts, master election, heartbeat status. Host failure events during ransomware attack chain visible. HA master election disruption can be used to prevent VM auto-recovery. Useful for reconstructing attack sequence in multi-host incidents.
esxupdate.log integrity /scratch/log/esxupdate.log Only if scratch configured ESXi patch and VIB (VMware Installation Bundle) install/remove events. Records every software change to the host. Malicious VIB installation (backdoor persistence mechanism — e.g. EvilVIB) appears here. Any CommunitySupported acceptance level VIB is suspicious. Cross-reference with esxcli software vib list output.
rhttpproxy.log network /var/log/rhttpproxy.log Only if scratch configured HTTP/HTTPS connections proxied to ESXi web services. All web-based management connections including NFC transfers. CVE exploitation against ESXi web services (e.g. CVE-2021-21985, CVE-2021-22005) leaves traces here. Source IPs of all web management connections visible.
Per-VM Artefacts (Datastore Level)
Artefact Path Persists reboot? Key forensic value Caveats
vmware.log vm log /vmfs/volumes/<datastore>/<VM name>/vmware.log Yes — on datastore Per-VM hypervisor log. Power on/off events, snapshot creation/deletion, vMotion source/destination, guest OS tools events, hardware configuration changes, VM suspend/resume. Rotated as vmware-1.log, vmware-2.log etc. (configurable retention). Survives VM deletion if datastore not wiped. Records Invoke-VMScript (CVE-2023-20867 exploitation) if VMware Tools is involved.
VMDK flat file (.vmdk / -flat.vmdk) disk image /vmfs/volumes/<datastore>/<VM name>/<VM>-flat.vmdk Yes — on datastore Full virtual disk of the VM. Forensic acquisition target — treat as physical disk image. Contains the entire guest OS filesystem. Acquire with dd or vmkfstools on the ESXi shell, or via FTK Imager over NFS mount. No write blocker available — alter timestamps of the VMDK container file but not internal disk data. Chain of custody documentation critical.
VM snapshot delta (.vmdk snapshots) snapshot /vmfs/volumes/<datastore>/<VM name>/<VM>-<N>.vmdk Yes — on datastore Differential disk changes since snapshot creation. Can preserve pre-attack state of VM disk if snapshot taken before encryption. Attacker-created snapshots may contain staging data. Attacker-created snapshots of VMs are a known data exfiltration staging technique — snapshot, export via NFC, delete snapshot. Check vmware.log and VI Events for unexpected snapshot creation. Snapshots listed via vim-cmd vmsvc/snapshot.getall <vmid>.
VMX file (.vmx) config /vmfs/volumes/<datastore>/<VM name>/<VM>.vmx Yes — on datastore VM configuration file. Hardware config, network adapters, ISO mounts, custom parameters. Changes leave no separate audit log — the file itself is the record. Attacker-modified VMX parameters (e.g. disabling VMware Tools, enabling serial port for C2, or changing network adapter) are visible by comparing current state to known-good baseline. Check guestinfo parameters for covert data.
VM memory snapshot (.vmem / .vmsn) memory /vmfs/volumes/<datastore>/<VM name>/ Yes — on datastore Memory state of a suspended or snapshotted VM. Contains full RAM content of the guest — credentials, encryption keys, network connections, running processes. .vmem = raw memory pages. .vmsn = snapshot state metadata. Analysable with Volatility (treat as raw memory dump). Only present if VM was suspended or snapshotted with memory included.

journalctl (VCSA systemd journal) collection
⚠ SECONDARY — collect after VMware logs		 OS-level service events, crashes, OOM kills, SSH sessions to the Photon OS appliance, unit start/stop — everything the VMware application logs do not capture at the OS layer Only after vpxd.log, VI Events, and sso-events.log have been reviewed. Escalate to journal when: attacker OS-level breakout is suspected, a VMware service crashed with no app-log record, or VCSA is being forensicated as a Linux host. journalctl --no-pager > /tmp/vcsa_journal.txt
journalctl -u vmware-vpxd --no-pager
journalctl -u vmware-vpostgres --no-pager
journalctl -u sshd --no-pager
journalctl --since "2024-10-01" --until "2024-10-21" --no-pager
journalctl -p err --no-pager		 Do not start here — vpxd.log and VI Events give better signal for typical vCenter attack scenarios (credential abuse, API calls, VM manipulation). Journal is the right next step when those logs show an unusual SSH session to the VCSA appliance itself, an unexplained vpxd crash, or evidence the attacker interacted with the Photon OS shell. Fire Ant / UNC3886 used OS-level access to the VCSA after vCenter exploitation — the journal was the primary evidence source for that post-exploitation phase. Cross-reference journal timestamps with vpxd.log and sso-events.log for a complete timeline.
vCenter support bundle collection All VCSA logs including vpxd, SSO, vpostgres, and appliance OS logs When vCenter itself is suspected to be compromised (not just managed hosts). DFIR4vSphere: Start-VC_Investigation -VCBundle
Or: https://<vcenter>:443/appliance/support-bundle		 Treat the VCSA as a Linux host for post-exploitation forensics. Standard Linux artefacts (bash_history, /etc/passwd, /var/log/auth.log) apply to the Photon OS appliance.
QELP — ESXi log parser analysis Parses support bundle logs into timeline CSV — SSH logins, shell enable/disable, user changes, file transfers, directory traversal Rapid triage of multiple ESXi hosts — surfaces attacker activity events quickly. QELP (Stroz Friedberg — open source CLI) Not all web client file access captured in syslog.log during testing. Combine with manual hostd.log and shell.log review for completeness.
esxcli / vim-cmd live queries live response Running processes, loaded VIBs, network connections, VM inventory, snapshot list Live host triage when the host is still running and accessible. esxcli software vib list
esxcli network ip connection list
vim-cmd vmsvc/getallvms
ps | grep -v grep		 VIB list is critical — any CommunitySupported or PartnerSupported VIB with an unusual name or recent install date (esxupdate.log) should be treated as suspect malicious VIB backdoor.
Remote syslog forwarding preventive All ESXi syslog output forwarded to external SIEM in real time Only effective if configured before the incident. Survives local log wiping. esxcli system syslog config set --loghost=udp://<siem>:514
esxcli system syslog reload		 If vmsyslogd is killed by attacker, forwarding also stops — but previously forwarded logs are preserved on the SIEM. This is why remote forwarding is the single most important defensive logging configuration for ESXi.
Known Threat Actor Techniques & Log Evidence
Technique What the attacker does Log evidence Residual artefacts
Malicious VIB installation persistence Installs a backdoor as a VIB (VMware Installation Bundle) on ESXi. VIBs survive reboots and can add files, services, and network listeners to the host. esxupdate.log: VIB install event with name, vendor, and acceptance level. Acceptance level CommunitySupported = no VMware signature validation. esxcli software vib list output collected in support bundle. VIB files persist in /tardisks/ and /locker/. VMware Verify_ESXi_VIB_signature.ps1 script checks signatures across all hosts. Known malicious VIBs: VIRTUALPITA, VIRTUALPIE (UNC3886 / Fire Ant).
vmsyslogd kill (log suppression) anti-forensics Terminates the vmsyslogd daemon to stop all local log writing and remote log forwarding simultaneously. Last entry in syslog.log before it goes silent. Remote syslog (if configured) shows the forwarding stop. Watchdog may attempt to restart the process — restart event in syslog if it recovers. Unexplained log gap is itself an IOC. If remote syslog was configured, gap in remote log stream timestamps the kill event precisely. Fire Ant (UNC3886) used this technique.
VCDB credential extraction credential access Queries the vCenter PostgreSQL database directly to extract host credentials from the vpx_host table (encrypted ESXi root passwords managed by vCenter). vpostgres/postgresql.log: LOG statement showing SELECT ip_address, user_name, password FROM vpx_host. vpxd.log may also show unusual database activity. Correlate with auth.log for the session that executed the query. Extracted credentials are AES-256 encrypted but decryptable with the vCenter symkey. Once decrypted, gives root access to all managed ESXi hosts. Document exactly which hosts' credentials were in the DB at time of breach.
SSH backdoor key planting persistence Adds attacker-controlled SSH public key to /etc/ssh/keys-root/authorized_keys for persistent passwordless root SSH access. auth.log: Subsequent logins show "Accepted publickey for root" with no preceding password failure. shell.log: Commands run in the backdoor session. hostd.log if the key was planted via API. Authorized_keys file survives reboot only if on persistent storage. Check /etc/ssh/keys-root/authorized_keys. On default ESXi (ramdisk only) the key must be re-planted after each reboot — scheduled task or VIB used to automate.
VM snapshot exfiltration exfiltration Creates a VM snapshot, exports the VMDK over NFC (Network File Copy), then deletes the snapshot to hide evidence. VI Events: vm.snapshot.create, vm.snapshot.remove in close succession. hostd.log: NFC connection establishment. vmkernel.log: NFC data transfer volume. vmware.log (per-VM): Snapshot create/delete timestamps. Even after snapshot deletion, the VI Events record the creation and deletion. vmware.log on the datastore preserves the events. SRUM-equivalent network telemetry on any connected Windows host may show the exfiltration volume.
Invoke-VMScript host-to-guest (CVE-2023-20867) lateral movement Uses PowerCLI Invoke-VMScript to execute commands inside guest VMs without guest authentication, bypassing in-guest security controls. vmware.log (per-VM): VMware Tools guest operation events. vpxd.log: Invoke-VMScript API call logged. hostd.log: Guest operation execution event. Requires VMware Tools installed in guest. Exploit of CVE-2023-20867 removes the authentication requirement. Used by Fire Ant/UNC3886. Patch VMware Tools and vCenter to remediate. Guest-side: process execution from vmtoolsd.exe parent with no interactive session is IOC.
Mass VM power off (pre-encryption) impact Powers off all VMs immediately before deploying the ESXi locker ransomware payload to ensure clean VMDK file locks for encryption. VI Events: Mass vm.powerOff events within seconds/minutes of each other from a single user account or API session. hostd.log: Rapid sequential power-off commands. vmkernel.log: Mass VM shutdown sequence. The time window between power-off events and the appearance of the ransomware locker binary on the host is often under 5 minutes. Timeline correlation between VI Events and shell.log shows the kill-then-encrypt sequence precisely.
