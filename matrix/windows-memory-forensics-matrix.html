Memory Analysis Methodology

A structured 8-phase approach for analysing a Windows memory dump. Click any phase to expand commands and hunting notes. Base syntax: vol.py -f memory.dmp <plugin>

1
Validate & profile the image
Confirm OS, build number, architecture before anything else
▼

Never skip this step. Wrong OS assumptions produce garbage output from every subsequent plugin. A truncated or corrupted dump will silently give you incorrect data. Confirm exactly what you are working with before interpreting any results.

Image identification
# OS version, build number, architecture, memory model vol.py -f memory.dmp windows.info # Verify image integrity — confirms dump is not truncated vol.py -f memory.dmp windows.memmap.Memmap # If auto-detection fails — list available symbol layers vol.py -f memory.dmp banners.Banners
  • Confirm kernel version matches expected build for the target environment
  • Note memory size — very small dumps may be partial crash dumps, not full RAM captures
  • Flag any mismatch between stated build and kernel structures discovered
2
Establish process baseline
What is running — and what is hidden
▼

Processes are always the starting point. Two views are required: the official EPROCESS linked list (pslist) and a raw pool scan (psscan). Discrepancies between these two are your first red flag — a process visible in psscan but absent from pslist has been unlinked, a classic rootkit technique. Build the full tree first, then identify anomalies before pivoting into individual processes.

Process enumeration
# Official EPROCESS linked list — what Windows knows about vol.py -f memory.dmp windows.pslist.PsList # Hierarchical parent/child tree — spot wrong-parent processes vol.py -f memory.dmp windows.pstree # Raw pool tag scan — catches unlinked/hidden processes (rootkits) vol.py -f memory.dmp windows.psscan.PsScan # Full command lines per process — catches cleared args, LOLBin abuse vol.py -f memory.dmp windows.cmdline.CmdLine # Scoped to a specific PID vol.py -f memory.dmp windows.cmdline.CmdLine --pid <PID> # Process token privileges — SeDebugPrivilege in unexpected process = red flag vol.py -f memory.dmp windows.privileges.Privs --pid <PID> # SIDs per process — spot SYSTEM SID in non-admin process (getsids IS in Vol3) vol.py -f memory.dmp windows.getsids.GetSIDs --pid <PID> # Multi-method process view — best for hiding detection vol.py -f memory.dmp windows.malware.psxview.PsXView
  • Diff pslist vs psscan — any PID in psscan missing from pslist has been DKOM-unlinked (rootkit)
  • lsass.exe: exactly one instance, parent must be wininit.exe, path must be System32
  • svchost.exe: all instances parent to services.exe, path System32 — any exception is suspicious
  • Flag: cmd.exe or powershell.exe spawned from Office apps, browsers, IIS, or Java
  • Flag: processes running from %TEMP%, %APPDATA%, ProgramData, or user-writable paths
  • Flag: misspelled process names — svch0st, lssas, csrrs, svchost32, explorer32
3
Deep-dive suspicious processes
DLLs, handles, threads, environment — per flagged PID
▼

Once Phase 2 yields suspicious PIDs, pivot into each one. You are hunting for three things: what modules are loaded (DLL injection), what resources the process holds (handles — files, pipes, mutexes), and what threads it has spawned (remote thread injection). The combination of all three tells the complete story about what the process is doing and how it was compromised.

Per-process investigation
# DLLs from PEB InLoadOrder list — baseline view vol.py -f memory.dmp windows.dlllist.DllList --pid <PID> # ldrmodules IS in Vol3 — compares 3 PEB loader lists to find hidden/unlinked DLLs vol.py -f memory.dmp windows.malware.ldrmodules.LdrModules --pid <PID> # FALSE in InLoad/InMem/InInit for a mapped region = reflective injection or hidden DLL # Open handles — files, registry keys, mutexes, named pipes, events vol.py -f memory.dmp windows.handles.Handles --pid <PID> # All handles across all processes (broader triage pass) vol.py -f memory.dmp windows.handles.Handles # Thread list — start addresses outside known modules = injection vol.py -f memory.dmp windows.threads.Threads --pid <PID> # Environment variables — staging paths, injected variables vol.py -f memory.dmp windows.envars --pid <PID> # Dump process executable to disk for static/AV analysis vol.py -f memory.dmp windows.dumpfiles.DumpFiles --pid <PID> --dump-dir ./dumps/
  • Mutex names are malware family fingerprints — search any unusual mutex name immediately
  • Named pipe handles in unexpected processes indicate C2 over SMB named pipe transport
  • DLL at unusual base address or loaded from non-System32/non-program path warrants extraction
  • Thread start address in private (non-module) memory region = classic shellcode injection
  • Any non-system process holding an open handle to lsass.exe = credential theft attempt in progress
4
Code injection detection
malfind, VAD analysis, shellcode — the irreplaceable phase
▼

This is where memory forensics is genuinely irreplaceable. Fileless malware, injected shellcode, process hollowing, and reflective DLL loads leave zero disk artifacts. The VAD tree is your map of a process's memory layout — regions that are executable but not backed by a file on disk are your primary target. malfind automates the most common checks. Always dump and run flagged regions through YARA or a sandbox before concluding.

Injection detection
# Find RWX memory not backed by a file — primary injection detection vol.py -f memory.dmp windows.malware.malfind.Malfind # Scoped to one process vol.py -f memory.dmp windows.malware.malfind.Malfind --pid <PID> # Dump all flagged regions for YARA/sandbox/reverse engineering vol.py -f memory.dmp windows.malware.malfind.Malfind --dump --dump-dir ./malfind_dumps/ # Direct hollowing detection vol.py -f memory.dmp windows.malware.hollowprocesses.HollowProcesses # Full VAD tree — complete virtual memory map for a process vol.py -f memory.dmp windows.vadinfo.VadInfo --pid <PID> # YARA scan against a specific process's VAD regions vol.py -f memory.dmp windows.vadyarascan --pid <PID> --yara-rules rules.yar # YARA scan across the entire memory image vol.py -f memory.dmp yarascan.YaraScan --yara-rules /opt/rules/malware.yar # SSDT hooks — syscall table tampering indicating kernel rootkit vol.py -f memory.dmp windows.ssdt # Kernel callbacks — registered notification routines (persistence/monitoring) vol.py -f memory.dmp windows.callbacks
  • RWX (read-write-execute) regions not mapped to a file on disk = almost always malicious
  • MZ/PE header (bytes 4D 5A) at the start of a private region = reflective DLL injection or process hollowing
  • Process hollowing signature: legitimate process with VAD showing private executable at the expected module base address
  • SSDT entries pointing outside ntoskrnl.exe or win32k.sys = active kernel-mode hook
  • Run all dumped regions through: strings → YARA → capa → VirusTotal hash lookup
5
Network artifact analysis
Active connections, closed sockets, C2 identification
▼

Network data read directly from kernel structures is far more reliable than a live netstat — rootkits commonly hook the userland APIs that netstat calls, but reading raw pool structures is much harder to tamper with. netscan will also surface connections that have recently closed but whose kernel objects haven't been freed, giving you a short historical window. Always correlate the owning PID against your process analysis from Phases 2 and 3.

Network connections
# Active TCP/UDP connections and listeners via TCPv4/v6 structures vol.py -f memory.dmp windows.netstat.NetStat # Pool scan — catches recently closed connections kernel hasn't freed yet vol.py -f memory.dmp windows.netscan.NetScan # Dump DNS client svchost memory to extract cached domain fragments vol.py -f memory.dmp windows.memmap.Memmap --dump --pid <DNS_svchost_PID> # Then extract domain strings from the dump # strings pid.<PID>.dmp | grep -iE '\.[a-z]{2,6}$' | sort -u > domains.txt
  • Any established connection from a process that should not have network access (notepad, calc, mspaint)
  • Outbound connections on 443/80 from non-browser processes — C2 traffic commonly blends with web traffic
  • Short-lived connections to rare external IPs — data exfil often bursts then immediately closes
  • Connections from unusual source to internal hosts on SMB (445), RDP (3389), WMI (135) = lateral movement
  • High-entropy or algorithmically-generated domain names in DNS fragments = DGA-based C2
  • Submit all external IPs to VirusTotal, Shodan, and AbuseIPDB for reputation check
6
Registry & persistence hunting
In-memory hives, run keys, services, deleted persistence
▼

Windows keeps all active registry hives mapped in memory. You can read registry data from the dump without ever touching disk — and critically, you may find persistence keys that have been deleted from disk but not yet flushed from the in-memory hive. This anti-forensics bypass catches attackers who clean up their tracks on disk: the deletion is on disk, but the in-memory copy hasn't been purged yet.

Registry persistence analysis
# List all registry hives loaded in memory vol.py -f memory.dmp windows.registry.hivelist # HKCU Run key — user-level autorun (most common malware persistence) vol.py -f memory.dmp windows.registry.printkey \ --key "SOFTWARE\Microsoft\Windows\CurrentVersion\Run" # RunOnce — executes once at next logon then self-deletes vol.py -f memory.dmp windows.registry.printkey \ --key "SOFTWARE\Microsoft\Windows\CurrentVersion\RunOnce" # Services key — all installed services including hidden ones vol.py -f memory.dmp windows.registry.printkey \ --key "SYSTEM\CurrentControlSet\Services" # Scan all service objects in memory pool (catches hidden services) vol.py -f memory.dmp windows.svcscan.SvcScan # UserAssist — GUI program execution history (ROT13 encoded subkeys) vol.py -f memory.dmp windows.registry.printkey \ --key "SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\UserAssist" # AppCompat ShimCache — execution evidence including deleted files vol.py -f memory.dmp windows.registry.printkey \ --key "SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache"
  • Registry keys pointing to executables in TEMP, AppData, ProgramData, or Downloads
  • Services with random or meaningless names, minimal descriptions, binary paths outside Program Files
  • svcscan returning more entries than the official service list = hidden service objects in pool
  • RunOnce is subtle — survives exactly one reboot then removes itself, easy to miss in disk forensics
  • UserAssist cross-referenced against process timeline can reveal execution gaps or tool usage
  • ShimCache contains evidence of files that executed even if the file was subsequently deleted
7
Credential extraction
LSASS, Kerberos, hashes, DPAPI — handle with care
▼

LSASS holds everything the attacker wants. The safest IR approach is to dump the LSASS process from Volatility and run Mimikatz offline against the dump on your analyst workstation — this avoids running any tooling live on the compromised host. Kerberos tickets are especially valuable: they are time-limited cryptographic tokens that cannot be reconstructed from disk and expire, so they must be captured from memory promptly.

Credential artifacts
# Direct NTLM hash extraction from SAM/LSASS kernel structures vol.py -f memory.dmp windows.registry.hashdump.Hashdump # LSA secrets — service account passwords, cached domain credentials ($DCC2) vol.py -f memory.dmp windows.registry.lsadump.Lsadump # Cached domain credentials ($DCC2 hashes) vol.py -f memory.dmp windows.registry.cachedump.Cachedump # Find lsass PID for process dump vol.py -f memory.dmp windows.pslist.PsList | grep lsass # Dump entire LSASS process for offline Mimikatz analysis vol.py -f memory.dmp windows.dumpfiles.DumpFiles --pid <lsass_PID> --dump-dir ./lsass_dump/ # Offline on analyst machine — extract logon passwords # mimikatz "sekurlsa::minidump lsass.dmp" "sekurlsa::logonpasswords" exit # Offline — export all Kerberos tickets to .kirbi files # mimikatz "sekurlsa::minidump lsass.dmp" "sekurlsa::tickets /export" exit # Clipboard — credentials, sensitive data, staged exfil content # No native clipboard plugin in Vol3 — dump the process owning the clipboard vol.py -f memory.dmp windows.memmap.Memmap --dump --pid <PID> -o ./clip_dump/ # Extract clipboard strings from process dump strings -el ./clip_dump/*.dmp | grep -iE ".{8,}" > clipboard_strings.txt # Console history — every command the attacker typed in cmd.exe vol.py -f memory.dmp windows.consoles.Consoles # Active logon sessions vol.py -f memory.dmp windows.sessions.Sessions
  • Kerberos tickets with 10-year validity = Golden Ticket — indicates full domain compromise
  • Service tickets without a corresponding TGT = Silver Ticket — targeted service forgery
  • WDigest enabled (UseLogonCredential=1 in HKLM) means cleartext passwords are in LSASS memory
  • NTLM hashes for domain admin or service accounts enable pass-the-hash lateral movement
  • Console history is especially valuable — it captures every command the attacker typed including recon, lateral movement, and cleanup commands
8
Extract & enrich artifacts
Dump, YARA scan, strings, threat intel — build the full picture
▼

The final phase converts memory artifacts into actionable intelligence. Dumped processes and memory regions go through static analysis, YARA rules, capability extraction (capa), and threat intel enrichment. The unpacked version of any packed malware found in memory is a significant analytical prize — it is ready for reverse engineering without fighting the packer, and often contains cleartext C2 configuration that the packed version hides.

Artifact extraction and enrichment
# Dump all files cached in memory by the Windows cache manager vol.py -f memory.dmp windows.dumpfiles.DumpFiles --dump-dir ./cached_files/ # Dump full address space of a specific process vol.py -f memory.dmp windows.memmap.Memmap --dump --pid <PID> # Dump malfind-flagged regions — unpacked malware, shellcode vol.py -f memory.dmp windows.malware.malfind.Malfind --dump --dump-dir ./injected/ # Scan entire image with YARA ruleset (Elastic, Signature-Base, custom) vol.py -f memory.dmp yarascan.YaraScan --yara-rules /opt/rules/malware.yar # File object scan — everything with an open file object in memory vol.py -f memory.dmp windows.filescan.FileScan # Kernel driver pool scan — diff with modules to find hidden drivers vol.py -f memory.dmp windows.driverscan.DriverScan vol.py -f memory.dmp windows.modules.Modules # Raw string extraction — IPs, URLs, domains, registry keys, paths # strings -el memory.dmp | grep -iE 'https?://|\\[A-Za-z]|HKEY' > strings_out.txt # Bulk IOC extraction (emails, URLs, IPs, domains, credit cards) # bulk_extractor -o bulk_out/ memory.dmp
  • Run all dumped PE files through sha256sum → VirusTotal lookup before opening locally
  • Use capa on dumped samples to extract capabilities (C2 comms, persistence, AV evasion) without full reversing
  • Drivers in driverscan not present in modules = hidden kernel module (active rootkit driver)
  • Unpacked malware dumps frequently contain cleartext C2 config — run strings first before reversing
  • Bulk IOC harvest from strings output: extract unique IPs, domains, file hashes → feed into SIEM detections
  • Submit extracted samples to ANY.RUN or Cuckoo for dynamic analysis and automated IOC extraction

Analyst Mental Model

Three parallel analysis lanes running simultaneously. The power is not in any single lane — it is in correlation across all three. A lone anomaly is noise. The same anomaly confirmed in two lanes is a lead. All three lanes pointing at the same PID or timestamp is your confirmed threat.

Why correlate across lanes?

Sophisticated attackers know that isolated indicators get dismissed. A single unusual process might be legitimate software behaving oddly. A connection to an odd IP might be telemetry or update traffic. A suspicious registry key might be leftover from software long uninstalled.

But when the same actor leaves a footprint in all three lanes simultaneously — a process running injected code (Process lane), making outbound C2 connections (Network lane), and installing a Run key for persistence (Persistence lane) — each lane validates the others and the case becomes unambiguous.

Process lane alone tells you something is executing. It does not tell you if it is exfiltrating data or whether it survives a reboot.
Network lane alone tells you there is suspicious traffic. Without knowing what process owns it, you cannot attribute or scope the compromise.
Persistence lane alone tells you the attacker intends to stay. It does not tell you what malicious code is currently active.

All three together tell the complete attack story: initial access → execution → persistence → command and control, all visible in a single memory snapshot. This is why correlation transforms weak signals into high-confidence, attribution-grade findings.

The golden rule of memory forensics

Memory analysis is about correlation and anomaly detection, not finding one smoking gun. A single weird process means little. A process with injected code, an active C2 connection, and a persistence mechanism is your case. Build the story across all artifacts and across all three lanes.

Memory is a snapshot in time. Its value degrades the moment the system reboots or the attacker runs cleanup. In IR, memory acquisition should happen before anything else — even before pulling event logs — because everything else persists to disk but memory does not. The analyst who gets the memory dump first gets the full picture.

Installation Guide

Step-by-step setup for Volatility 3 on Windows and Linux, including Python requirements, symbol tables, and recommended supporting tools.

1 — Prerequisites

Volatility 3 requires Python 3.8 or later. Most modern Linux distributions ship with a suitable version. Verify before proceeding.

Check Python version
# Must be 3.8 or higher python3 --version # Install pip if missing sudo apt install python3-pip python3-venv -y # Debian/Ubuntu sudo dnf install python3-pip -y # Fedora/RHEL
2 — Clone and install Volatility 3

The recommended approach is to clone from GitHub and install into a Python virtual environment to keep dependencies isolated from your system Python.

Clone and set up
# Clone the repository git clone https://github.com/volatilityfoundation/volatility3.git cd volatility3 # Create and activate a virtual environment (recommended) python3 -m venv venv source venv/bin/activate # Fix ownership if cloned to /opt as root (common on Kali) sudo chown -R $USER:$USER /opt/volatility3 # Install Volatility 3 and all dependencies (no sudo needed after chown) pip install -e ".[full]" # Verify installation python3 vol.py --version
Alternative — pip install (no git clone):
pip install volatility3 installs the latest release from PyPI. The GitHub clone is preferred for DFIR work as it includes the latest plugin updates between releases.
3 — Symbol tables (ISF files)

Volatility 3 requires Intermediate Symbol Files (ISF) to parse kernel structures. Without the correct symbol table for the target OS, most plugins will fail. There are two ways to get them.

Option A — Download pre-built symbol packs (recommended)
# Create the symbols directory mkdir -p volatility3/symbols # Download Windows symbol pack from Volatility Foundation # https://downloads.volatilityfoundation.org/volatility3/symbols/windows.zip wget https://downloads.volatilityfoundation.org/volatility3/symbols/windows.zip unzip windows.zip -d volatility3/symbols/ # Also grab Linux and macOS packs if needed wget https://downloads.volatilityfoundation.org/volatility3/symbols/linux.zip wget https://downloads.volatilityfoundation.org/volatility3/symbols/mac.zip unzip linux.zip -d volatility3/symbols/
Option B — Auto-download at runtime
# Vol3 can attempt to fetch the right symbol table automatically # when it identifies the image — requires internet access python3 vol.py -f memory.dmp windows.info # If auto-download fails, Vol3 will print the exact ISF URL to fetch manually
Symbol table location: Vol3 searches volatility3/symbols/ by default. You can also set the environment variable VOLATILITY_SYMBOL_PATH to point to a custom directory, useful when maintaining a central symbol store across multiple installations.
4 — Recommended supporting tools

Volatility alone covers the analysis side. These tools handle acquisition, post-extraction analysis, and IOC enrichment.

Quick smoke test

6 — Set up the vol command (skip typing python3 every time)

Three options — pick whichever suits your workflow. The wrapper script is the most robust as it survives shell reloads and works across all terminals.

Option A — Shell alias (quick, per-session or persistent)
# Add to ~/.bashrc or ~/.zshrc for persistence echo 'alias vol="source /opt/volatility3/venv/bin/activate && python3 /opt/volatility3/vol.py"' >> ~/.bashrc source ~/.bashrc # Now just run: vol -f memory.dmp windows.info
Option B — Wrapper script in /usr/local/bin (recommended)
# Create the wrapper script sudo tee /usr/local/bin/vol > /dev/null << 'EOF' #!/bin/bash source /opt/volatility3/venv/bin/activate exec python3 /opt/volatility3/vol.py "$@" EOF sudo chmod +x /usr/local/bin/vol # Verify it works from any directory vol --version vol -f /path/to/memory.dmp windows.info
Option C — Symlink (only works if venv is always active)
# Only use this if you always activate the venv before working sudo ln -s /opt/volatility3/venv/bin/python3 /usr/local/bin/vol-python # Then set alias to use the venv python directly echo 'alias vol="vol-python /opt/volatility3/vol.py"' >> ~/.bashrc
Recommended: Option B (wrapper script) is the most reliable. It activates the venv automatically every time, works from any directory, survives terminal restarts, and does not require you to remember to run source activate first. After setting it up, all examples in this reference that show python3 vol.py can be run as just vol.

7 — Set up the vol command on Windows

On Windows you can create a batch file wrapper so you can type vol from any directory instead of specifying the full python path each time.

Option A — Batch file wrapper (PowerShell / cmd)
# Create a vol.bat in a folder already on your PATH (e.g. C:\Windows\System32 or C: ools) # Run this in PowerShell as Administrator: $bat = @' @echo off call C:\opt\volatility3 env\Scriptsctivate.bat python C:\opt\volatility3\vol.py %* '@ $bat | Set-Content "C:\Windows\System32\vol.bat" # Now from any cmd or PowerShell window: vol -f C:\cases\memory.dmp windows.info
Option B — PowerShell alias (persistent via profile)
# Add a persistent function to your PowerShell profile Add-Content $PROFILE 'function vol { & "C:\opt\volatility3\venv\Scripts\python.exe" "C:\opt\volatility3\vol.py" @args }' # Reload profile . $PROFILE # Now just run: vol -f C:\cases\memory.dmp windows.pslist.PsList
Note: Adjust the paths (C:\opt\volatility3) to wherever you cloned Volatility. Option B (PowerShell function) is cleaner for PowerShell-only workflows; Option A (batch file) works in both cmd and PowerShell.
Common issues & fixes
Error / symptomCauseFix
No suitable address space mapping found Missing or wrong symbol table for the target OS build Download the correct Windows symbol pack and place in volatility3/symbols/
ImportError: No module named 'yara' yara-python not installed in active venv pip install yara-python
Plugin returns zero results Wrong plugin for the OS, or symbol table mismatch Run windows.info first — confirm OS and build, then re-run plugin
Unsatisfied requirement windows.KernelModule Vol3 cannot locate the kernel in the dump — often a partial or hibernation file Ensure the dump is a full RAM image, not a crash dump or hiberfil.sys
Very slow on large dumps (>32GB) Vol3 rebuilds its internal page map on each run by default Add -v to see progress; consider --cache options for repeated runs on the same image
windows.hashdump returns no results SAM hive not loaded or SYSTEM hive missing from dump Run windows.registry.hivelist to confirm both SAM and SYSTEM hives are present
Plugin not found / no such plugin Using a deprecated or renamed plugin (e.g. windows.malfind → windows.malware.malfind, windows.hashdump → windows.registry.hashdump, connections/apihooks are Vol2 only) See the Vol3 equivalents documented throughout this reference — many Vol2 plugins were renamed or replaced
